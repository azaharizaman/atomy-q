# Atomy-Q Identity Gap 7 MFA and Audit Extension Design

**Topic:** Complete the remaining Gap 7 identity work for alpha by finishing MFA verification and replacing the no-op audit sink with persisted audit logging.
**Date:** 2026-04-08
**Status:** Draft

## 1. Executive Summary
The base Gap 7 spec already calls for MFA, server-side sessions, RBAC, and audit logging. The current worktree has the RBAC/session/login lockout slice implemented, but two production gaps remain:

1. `POST /auth/mfa/verify` still returns a stub response.
2. Audit writes still disappear into a no-op repository instead of being persisted.

This extension finishes those two gaps without reopening the already-completed RBAC/session work. The MFA part adds a real challenge/verify flow that bridges the current HTTP `challenge_id` contract to the existing Nexus MFA verification service. The audit part swaps the no-op repository for a DB-backed adapter and persists identity events generated by the auth flow.

## 2. Goals & Success Criteria
- **Goal:** Make `POST /auth/mfa/verify` a real verification endpoint that can complete login for users requiring TOTP or backup-code MFA.
- **Goal:** Persist identity audit records in the database instead of dropping them on the floor.
- **Success Criteria:**
  - `POST /auth/login` returns either a normal JWT response or an MFA challenge response, depending on the user state.
  - `POST /auth/mfa/verify` accepts a real `challenge_id` and `otp`, verifies the challenge, and issues usable authenticated tokens.
  - MFA challenges are single-use and expire.
  - Audit records are written for login success, login failure, MFA challenge issuance, MFA verification success/failure, and logout.
  - The existing RBAC, lockout, and session revocation behavior remains unchanged.

## 3. Scope Boundaries
This extension is intentionally narrow.

- **In scope:** MFA challenge issuance and verification, audit log persistence, controller wiring, adapter binding, and regression tests.
- **Out of scope:** New MFA enrollment UI, WebAuthn/passkey support, SSO changes, audit browsing UI, retention policy tuning, and broader identity package refactors not needed to complete the two missing alpha behaviors.

## 4. Architecture & Data Flow

### 4.1 MFA Verification
The current orchestrator already knows how to verify a user ID, MFA method, and OTP. What is missing is the alpha-facing bridge between the login endpoint and that verifier.

The extension introduces a small app-local MFA challenge store:

- `login` authenticates the password as it does today.
- If the account requires MFA, the auth layer creates a one-time challenge record and returns `401 MFA_REQUIRED` with `challenge_id`.
- `mfaVerify` resolves the challenge ID, confirms it belongs to the same tenant and is not expired or consumed, and then delegates OTP validation to the existing MFA coordinator.
- On success, the challenge is consumed and the final authenticated tokens are issued.

This keeps the challenge opaque to the client while still preserving tenant scope and replay protection.

### 4.2 Audit Persistence
The identity adapter already calls `AuditLogRepositoryInterface::create()`, but the current app binding points that interface at a no-op repository. The extension replaces that binding with a real Laravel adapter backed by an `audit_logs` table.

Audit writes stay inside the existing identity flow:

- auth success and failure events are written from the login flow,
- MFA challenge issuance and verification are written from the MFA bridge,
- logout writes continue to be emitted through the coordinator and are now persisted,
- the adapter stores the structured event payload so downstream reporting can query it later.

## 5. Data Model

### 5.1 MFA Challenges
Add a dedicated `mfa_challenges` table for one-time login challenges.

Recommended columns:
- `id` string primary key, random opaque challenge identifier
- `user_id` ULID, indexed
- `tenant_id` ULID, indexed
- `method` string, for alpha this is `totp` or `backup_codes`
- `expires_at` timestamp, indexed
- `consumed_at` timestamp, nullable
- `attempt_count` integer, default `0`
- `created_at` / `updated_at`

This table is used only for pending MFA login challenges. It is not the MFA enrollment store.

### 5.2 Audit Logs
Add a persisted `audit_logs` table compatible with the existing `AuditLogRepositoryInterface` contract.

Recommended columns:
- `id` ULID primary key
- `event` string
- `subject_type` string, nullable
- `subject_id` string, nullable
- `causer_type` string, nullable
- `causer_id` string, nullable
- `tenant_id` ULID, nullable
- `level` integer
- `batch_uuid` string, nullable
- `properties` json
- `created_at` / `updated_at`

The schema should preserve the structured payload emitted by `IdentityOperationsAdapter::log()` so the adapter can round-trip the data without lossy translation.

## 6. Component Changes

### 6.1 Authentication Flow
The authentication service needs one additional decision point before it creates final tokens:

- if the user does not require MFA, it behaves as it does today;
- if the user requires MFA, it throws `MfaRequiredException` before token issuance;
- the controller catches that exception, creates the `challenge_id`, and returns the MFA-required response.

This keeps password verification in the existing auth path while deferring token issuance until MFA is completed.

### 6.2 MFA Verification Endpoint
`POST /auth/mfa/verify` becomes a two-step bridge:

1. Validate and load the challenge by `challenge_id`.
2. Verify the `otp` using the existing MFA coordinator/service for the challenge user and method.
3. Consume the challenge.
4. Call the coordinator completion path that issues the final session-backed tokens.

The endpoint must not leak whether the challenge or user exists beyond the authenticated tenant context.

### 6.3 Audit Adapter
The audit repository adapter persists the payload using Eloquent instead of throwing or returning empty results. The adapter must keep the existing interface shape so the identity adapter can continue to call `create()` without special handling.

The app provider should stop binding the audit contract to the no-op repository and bind the persisted adapter instead.

## 7. Error Handling
- Invalid or expired MFA challenges return `401` with a generic failure message.
- Reused or already-consumed challenges are rejected.
- Wrong OTP values increment challenge attempts and do not mint tokens.
- Audit persistence failures should be reported, but they must not crash the auth endpoint unless the failure prevents completion of a success path that must be recorded.

## 8. Testing Strategy
- Verify login returns `MFA_REQUIRED` when the account requires MFA.
- Verify `mfaVerify` succeeds with a valid challenge and OTP and returns final tokens.
- Verify expired, consumed, or tampered challenges fail.
- Verify backup-code MFA succeeds through the same verification path.
- Verify audit entries are persisted for login success, login failure, MFA challenge issuance, MFA success, and logout.
- Verify tenant isolation by using a challenge from a different tenant and expecting rejection.
